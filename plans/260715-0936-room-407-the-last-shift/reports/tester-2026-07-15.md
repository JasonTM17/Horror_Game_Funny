## Resolved findings from first run

### Memory pickup branch never matched

The first run found a GDScript array pattern that could not match any memory string. It was replaced with explicit string cases plus `_collect_memory()`. The progression test now collects `memory_photo`, `memory_cassette`, and `memory_rabbit`, then rejects a duplicate.

### Solid back wall blocked the chase route

`Room407Back` was moved to z `-140`, past the exit. The final clue → chase threshold → exit route is now spatially open. Manual collision/feel validation remains required because headless smoke does not drive a player through the route.

### Duplicate chase entity after failure

`fail_chase()` now restores and reasserts the chase flag before restarting the existing entity. The progression test counts `TheEntity*` children after failure and passes with one entity.

### Checkpoint scene reload reset

`GameplayDirector._ready()` now resets only when no checkpoint exists, preserving state when a scene-based checkpoint reload is used. A dedicated router reload test remains desirable.

## Remaining concerns

- Phase 7's broad test matrix still has only the GameState and progression scenes. Focused interaction/puzzle/checkpoint/settings suites are not present yet; the new runner now covers import/menu/gameplay plus both available suites.
- The progression test now calls `get_tree().quit(2)` on failure, so this suite no longer produces a false zero exit. The GameState test still relies on `assert()` and should use the same explicit failure helper for CI reliability.
- Dynamic hallway currently changes labels, not variant roots/collision/process state described by the architecture contract; manual review should confirm whether this is an accepted simplification.
- The progression test exits cleanly but reports `WARNING: 24 ObjectDB instances were leaked at exit`; this is teardown noise, not an assertion failure, but should be cleaned before release.
- Running `progression-test.gd` with `--scene` passes deterministically in three consecutive runs. The committed runner correctly uses the scene wrapper; direct `--script` mode still fails during compile because preloaded gameplay scripts resolve `GameState`/`SettingsManager` before autoloads are available.
- No authored pacing timers/cinematics are present in the continuous director; only 4–6 second horror prop timers and a 1.25 second failure overlay. The 139 m straight path takes roughly 53 seconds at walk speed (about 32 seconds sprint), so the 15–20 minute target needs a measured blind run or additional authored observation/puzzle beats.
