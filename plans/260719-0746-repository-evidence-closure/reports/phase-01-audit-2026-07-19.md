## Static findings

### Blocker B1 — pre-launch unique side-channel can pass freshness

`tests/run-physical-playthrough.ps1:169` accepts a file written as early as two seconds
before `$NotBeforeUtc`; that boundary is recorded only after `Prepare-...` returns at
lines 408-410. A uniquely hashed file created after the archive/clear scan but before
that timestamp (or timestamped within the tolerance) is neither an archived baseline
nor older than the threshold, so it is copied and can supply the sole payload. The
regression tests an old file at minus five minutes, not this interval. This contradicts
the stronger “prevents a previous run” language in README/testing/limitations.

### Blocker B2 — path/hash/size evidence checks are check-then-use operations

`Get-PacingEvidenceRecord` checks the leaf/reparse attribute, then obtains hash/size;
later `Copy-Item` reopens the source by pathname and only compares a subsequent source
read to the copy (`run-physical-playthrough.ps1:111-120, 176-205`). A concurrent writer
or directory/file reparse swap can change the resolved object between those operations.
The static junction cases prove rejection of an already-present junction, not freshness,
object identity, or containment across this race. This is material because the payload
is an evidence side-channel, even though it can never prove human input by itself.

### Warning W1 — Windows containment/preservation remains unproved in this audit

The new adversarial harness statically records active/rollback identities before and
after timeout and lock probes, and its timeout occurs before verifier publication.
That is sensible coverage, but it was not run here. In addition, the existing
`tests/windows-export-transaction.ps1` uses `Assert-NoReparsePointPath` before later
`Remove-Item`/`Directory.Move` pathname operations; a Windows reparse swap after the
check is not closed by this new harness. Therefore Phase 1 cannot prove negative tests
cannot mutate accepted bundles under adversarial concurrent filesystem control.

### Warning W2 — result wording is historical, not current verification

The edited 2026-07-18 QA report says both new harnesses passed and shows a bundle hash,
but this audit did not locate or rerun contemporaneous raw evidence. Treat those claims
as historical assertions only. Likewise `docs/testing.md` describes a 2026-07-18 pass;
neither statement can satisfy Phase 3's fresh verification requirement.
